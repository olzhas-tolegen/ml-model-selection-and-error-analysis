# Data Policy

## Purpose

This document defines how data is handled in the `multiclass-model-selection-and-error-analysis` repository.

The goal is to keep the public repository clean, lightweight, reproducible, and compliant with responsible data-management practices.

## Summary

Raw datasets, processed datasets, generated metrics, model artifacts, and notebook outputs are intentionally excluded from Git.

The repository should track:

- Source notebooks
- Python source code
- Scripts
- Documentation
- Environment files
- Folder placeholders such as `.gitkeep`

The repository should not track:

- Raw datasets
- Processed datasets
- Generated CSV files
- Generated figures
- Model binaries
- Notebook execution outputs
- Cache files
- Temporary files

## Why Data Is Not Committed

Datasets and generated artifacts are excluded for several reasons:

1. **Repository size control**  
   ML datasets and generated outputs can quickly make a Git repository large and slow.

2. **Reproducibility discipline**  
   Processed files should be regenerated from documented scripts and notebooks.

3. **Licensing caution**  
   Public redistribution of third-party datasets may be restricted or ambiguous.

4. **Cleaner portfolio review**  
   Recruiters and engineering reviewers should see source code, documentation, and reproducibility instructions, not large generated artifacts.

5. **Better Git history**  
   Generated files create noisy diffs and make version history harder to review.

## Expected Local Data Layout

Each project module may define its own local data layout. For the first module, the expected structure is:

```text
multiclass-model-selection-and-error-analysis/
└── data/
    ├── raw/
    │   └── covertype/
    │       └── covtype.data
    └── processed/
        ├── dataset_stratified.csv
        └── dataset_remainder.csv
```

The raw dataset should be downloaded or placed locally by the user. It should not be committed to Git.

## Processed Data

Processed datasets are generated by the project workflow.

For the first module:

```text
dataset_stratified.csv
dataset_remainder.csv
```

These files are generated locally and are ignored by Git.

The intended protocol is:

```text
dataset_stratified.csv
├── 80% training
└── 20% validation

dataset_remainder.csv
└── untouched final test set
```

The processed data files must be regenerated locally when needed.

## Results and Figures

Generated results and figures are also excluded from Git by default.

Ignored output locations include:

```text
results/metrics/
results/runtime/
results/model-comparison/
results/confusion-matrices/

figures/eda/
figures/learning-curves/
figures/model-complexity-curves/
figures/confusion-matrices/
figures/runtime/
```

These directories are preserved with `.gitkeep` files so the folder structure remains visible on GitHub.

## Model Artifacts

Serialized models should not be committed unless there is a specific reason to publish a small final artifact.

Ignored model formats include:

```text
*.pkl
*.pickle
*.joblib
*.pt
*.pth
*.onnx
*.h5
*.ckpt
```

For this portfolio repository, model binaries are not required. The preferred approach is to regenerate models from the notebooks and scripts.

## Notebook Outputs

Notebook outputs should be cleared before committing.

Before pushing to GitHub, run:

```bash
cd multiclass-model-selection-and-error-analysis
bash scripts/clean_outputs.sh
```

This removes:

- Notebook execution outputs
- Execution counts
- Generated metrics
- Generated figures
- Processed datasets
- Cache files
- Temporary files

## `.gitkeep` Policy

Git does not track empty directories. This repository uses `.gitkeep` files to preserve important folder structure.

Examples:

```text
data/raw/covertype/.gitkeep
data/processed/.gitkeep
results/metrics/.gitkeep
figures/eda/.gitkeep
```

These files are intentionally committed.

## `.gitignore` Policy

The repository `.gitignore` should exclude generated and local-only files while explicitly allowing important source files.

Allowed:

```text
README.md
portfolio-summary.md
data-policy.md
environment.yml
requirements.txt
reproducibility.md
notebooks/*.ipynb
src/**/*.py
scripts/*.sh
reports/*.md
.gitkeep
```

Ignored:

```text
data/raw/*
data/processed/*
results/*
figures/*
__pycache__/
.ipynb_checkpoints/
.venv/
*.csv
*.pkl
*.pt
```

## Reproducibility Expectation

A reviewer should be able to reproduce results by:

1. Cloning the repository
2. Creating the environment
3. Placing the raw dataset in the documented local path
4. Running the project scripts
5. Regenerating processed datasets, metrics, and figures locally

Example:

```bash
cd 01-model-benchmarking-and-error-analysis
conda env create -f environment.yml
conda activate ml-model-evaluation
bash scripts/run_all.sh
```

## Public Repository Rule

The public repository should answer three questions clearly:

1. What does the project do?
2. How can it be reproduced?
3. What data is required but intentionally not committed?

This document exists to make that policy explicit.

## Recommended Pre-Push Checklist

Before pushing:

```bash
bash scripts/clean_outputs.sh
git status
```

Confirm that Git is tracking only:

- Markdown documentation
- Notebooks with cleared outputs
- Python source files
- Shell scripts
- Environment files
- `.gitignore`
- `.gitkeep`

Confirm that Git is not tracking:

- Raw data
- Processed data
- Generated metrics
- Generated figures
- Notebook checkpoints
- Python caches
- Virtual environments
- Model binaries

## Final Policy

This repository is source-controlled as a reproducible ML project, not as a storage location for datasets or generated artifacts.

Data and outputs are regenerated locally. Source code, notebooks, scripts, and documentation are committed publicly.
